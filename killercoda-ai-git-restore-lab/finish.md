# 解説

この演習の答えは、退避されていた `.git_disabled` を `.git` に戻すことです。

```bash
mv .git_disabled .git
```

## なぜこれで直るのか

Git は作業ツリーのルートにある `.git` ディレクトリを使って、履歴やブランチ、インデックスなどの管理情報を保持します。
そのため `.git` が消えると、同じフォルダにコードが残っていても、Git からは「ただの普通のフォルダ」に見えてしまいます。

今回のように `.git` が `.git_disabled` に退避されている場合は、管理情報を元の名前へ戻せば Git が再び作業ツリーを認識します。

## 受講者に伝えたいポイント

- 問題の本質はコードではなく、ワークスペースのメタデータ破損だった
- 隠しファイルの確認は、開発環境のトラブル調査で有効
- AI 支援ツールの安全機能は便利だが、ワークスペースの重要な状態を変えていないか確認が必要
- Git 管理情報は、アプリの設定ファイルと同じくらい慎重に扱うべき

## 期待される確認結果

```bash
cd /root/lab/workspace
git status
```

`git status` が実行でき、作業ツリーがクリーンなら復旧成功です。
